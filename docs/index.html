🔍 Pulsar Batch Acknowledgment Issue - Root Cause Analysis

SEVERITY: CRITICAL - Production Duplicate Processing

Investigation Date: March 16, 2026

Issue: Duplicate message processing in Pulsar consumer workers despite successful acknowledgment

Impact: Merchant balance updates processed multiple times, causing incorrect balance calculations

📊 Executive Summary

Root Cause Confirmed: Pulsar broker configured with acknowledgmentAtBatchIndexLevelEnabled=false or batchIndexAckEnabled=false tracks acknowledgments at the batch level instead of individual message level. When any message in a producer batch is negatively acknowledged, the entire batch is marked as unacknowledged and redelivered to another worker, causing duplicate processing of already-acknowledged messages.
Fix Verified: Enable acknowledgmentAtBatchIndexLevelEnabled=true on the broker and batchIndexAckEnabled=true on consumers. This enables per-message acknowledgment tracking within batches, ensuring only negatively acknowledged messages are redelivered.

🔬 Root Cause Analysis

The Problem: Batch-Level Acknowledgment Tracking

When acknowledgmentAtBatchIndexLevelEnabled=false or batchIndexAckEnabled=falses on the Pulsar broker, the broker tracks acknowledgments at the batch level, not individual message level. This causes the following behavior:

Producer sends 5 messages
→
Batched into single batch
→
Worker 1 receives batch
↓
ACK: msg-1,2,4,5
NACK: msg-3
→
Broker marks ENTIRE BATCH as unacked
→
ALL 5 messages redelivered

Why This Happens

  1. Producer Batching: Pulsar producer batches multiple messages together for efficiency (configured with batchingEnabled: true)
  2. Broker Storage: The batch is stored as a single entry in BookKeeper ledger
  3. Batch-Level Cursor: Without acknowledgmentAtBatchIndexLevelEnabled, the broker's managed cursor tracks acknowledgment status at the batch entry level, not individual messages
  4. Negative Acknowledgment: When msg-3 is negatively acknowledged, the broker marks the entire batch entry as "unacknowledged"
  5. Redelivery: The entire batch is redelivered to the next available worker (or same worker after timeout)

Simulation vs Production Timing Difference

Environment Redelivery Interval Trigger Mechanism Explanation
Simulation ~3.3 seconds Batch receive timeout Consumer's batchReceivePolicy.timeout: 10000ms expires while waiting for next batch. Pulsar immediately redelivers unacked messages to fill the batch.
Production ~60 seconds Negative ack redelivery delay Pulsar's default negativeAckRedeliveryDelay is 60 seconds. After this delay, the entire batch is redelivered to another worker.
Key Insight: The simulation shows continuous redelivery every ~3 seconds because the consumer immediately requests the next batch after processing, triggering rapid redelivery of unacked messages. In production, the 60-second gap is caused by Pulsar's default negativeAckRedeliveryDelay (60 seconds), which was not explicitly configured in the consumer. After a message is negatively acknowledged, Pulsar waits exactly 60 seconds before making the entire batch available for redelivery to another worker.

🔧 Technical Deep Dive

How Batch Index Acknowledgment Works

Based on PR #6052 (PIP-54: Support acknowledgment at batch index level):

// WITHOUT batch index ack (batchIndexAckEnabled=false, acknowledgmentAtBatchIndexLevelEnabled=false) Broker Cursor Tracking: Entry (ledgerId:123, entryId:456) → Status: UNACKED (entire batch) ↓ When msg-3 is NACKed: - Entire entry marked as unacked - All 5 messages redelivered // WITH batch index ack (batchIndexAckEnabled=true, acknowledgmentAtBatchIndexLevelEnabled=true) Broker Cursor Tracking: Entry (ledgerId:123, entryId:456) → BitSet: [0:ACK, 1:ACK, 2:NACK, 3:ACK, 4:ACK] ↓ When msg-3 is NACKed: - Only index 2 marked as unacked - Only msg-3 redelivered

❓ Why Duplicates Don't Always Happen

The duplicate processing issue is intermittent in production because it depends on specific conditions being met:

Simulation vs Production Environment

Factor Simulation Production Impact on Duplicates
Number of Workers 1-2 workers 10+ workers More workers = higher chance different worker receives redelivered batch
Batch Distribution All messages in same batch Messages spread across multiple batches Only messages in the SAME producer batch are affected
Failure Rate 100% (msg-3 always fails) Low (~0.1-1% of messages) Duplicates only occur when at least one message in a batch fails
Worker Assignment Same worker often receives redelivery Load balanced across 10+ workers Redelivered batch likely goes to different worker with no deduplication state

Conditions Required for Duplicate to Occur

  1. Messages must be in the same producer batch:
    • Producer sends messages with batchingEnabled: true
    • Multiple proposals sent close together are batched
    • Example: Proposals A, B, C, D, E sent within 10ms → single batch
  2. At least one message in the batch must fail:
    • OCC failure (version mismatch)
    • Validation error
    • Network timeout
    • Database error
    • → Triggers negativeAcknowledge()
  3. Redelivered batch must be picked up by a different worker:
    • With 10+ workers, high probability of different worker
    • Different worker = no shared client-side deduplication state
    • → All messages in batch processed again

Why It's Intermittent

Example Scenario:

  • Batch 1: 4000 proposals, all succeed → No duplicates
  • Batch 2: 4000 proposals, 1 fails (OCC error) → All 4000 redelivered after 60s
  • Batch 3: 4000 proposals, all succeed → No duplicates
  • Batch 4: 4000 proposals, 2 fail → All 4000 redelivered after 60s

Result: Duplicates only appear when a batch contains at least one failure. With a low failure rate (~0.1%), most batches process successfully without duplicates.

🎯 Solution Implementation

1. Broker Configuration (Critical)

# In broker.conf or via environment variable acknowledgmentAtBatchIndexLevelEnabled=true

For Docker Compose:

broker: environment: - acknowledgmentAtBatchIndexLevelEnabled=true

2. Consumer Configuration (Recommended)

Update your consumer subscribe options:

consumer = await client.subscribe({ topic: 'tx1', subscription: 'tx2', subscriptionType: 'Shared', ackTimeoutMs: USER_ACK_TIMEOUT_MS ? USER_ACK_TIMEOUT_MS : derivedAckTimeoutMs, batchIndexAckEnabled: true, // ✅ ADD THIS LINE - Enables per-message ack tracking negativeAckRedeliveryDelay: 60000, // ✅ OPTIONAL - Make default explicit (60s) receiverQueueSize: BATCH_SIZE, batchReceivePolicy: { maxNumMessages: BATCH_SIZE, maxNumBytes: 1024 * 1024 * 10, timeoutMs: TIMEOUT_MS } });

Note: The negativeAckRedeliveryDelay parameter is optional (defaults to 60000ms). Making it explicit improves code clarity when analyzing redelivery timing.

3. Verification Steps

  1. Enable acknowledgmentAtBatchIndexLevelEnabled=true on production broker
  2. Add batchIndexAckEnabled: true to consumer configuration
  3. Deploy changes
  4. Monitor logs for duplicate processing
  5. Verify only negatively acknowledged messages are redelivered

✅ Recommended Actions

Immediate Actions (Production)

  1. Enable broker-side batch index acknowledgment:
    • Set acknowledgmentAtBatchIndexLevelEnabled=true in broker configuration
    • Restart broker (rolling restart for zero downtime)
  2. Enable consumer-side batch index acknowledgment:
    • Add batchIndexAckEnabled: true to consumer subscribe options
    • Deploy updated worker code
  3. Monitor for duplicates:
    • Check logs for duplicate proposalId processing
    • Verify only negatively acknowledged messages are redelivered
    • Monitor merchant balance accuracy

Additional Recommendations

  1. Idempotency: Implement idempotent processing logic using the existing OCC (version field) to handle any remaining edge cases
  2. Dead Letter Queue: Configure DLQ for messages that fail repeatedly after negative acknowledgment
  3. Monitoring: Add metrics for duplicate detection and negative acknowledgment rates
  4. Alerting: Alert when duplicate processing is detected (even with fix, edge cases may exist)

📝 Summary

Problem: Pulsar broker with acknowledgmentAtBatchIndexLevelEnabled=false tracks acknowledgments at batch level, causing entire batches to be redelivered when any message is negatively acknowledged.

Impact: Successfully processed and acknowledged messages are redelivered and processed again, causing duplicate merchant balance updates.

Solution: Enable batch index acknowledgment on both broker (acknowledgmentAtBatchIndexLevelEnabled=true) and consumer (batchIndexAckEnabled=true) to track acknowledgments at individual message level within batches.

Verification: Simulation confirms the issue is reproducible with broken configuration and fixed with proper configuration. Production logs show 60-second redelivery pattern consistent with negative acknowledgment redelivery delay.

Investigation completed: March 16, 2026
Pulsar versions tested: 2.6.4 (bug reproduced), 3.0.0 (bug present only with explicit managed ledger config disabled)
Note: Pulsar 3.0.0 has additional managed ledger layer protection (deletionAtBatchIndexLevelEnabled=true by default) that prevents the issue unless explicitly disabled. Pulsar 2.6.4 was used for clean reproduction.
